Tu es un expert Supabase (Postgres + RLS) et tu construis le système de paiement “Stratégie B” pour SignTouch :

Objectif
- Les fans achètent des sessions live via Apple IAP / Google Play.
- L’app enregistre et valide l’achat (transaction_id / purchase_token).
- On NE paie la célébrité qu’après import des rapports financiers Apple/Google (net bancaire exact).
- Supabase doit stocker : gross, status, et ensuite net_final (proceeds exact), puis calculer la part célébrité et préparer un batch de payout.

Contraintes
- Multi-plateforme : apple / google.
- 1 achat = 1 session live (consumable).
- On doit gérer refunds/chargebacks.
- Les fans ne doivent voir QUE leurs propres sessions/transactions.
- Les célébrités ne doivent voir QUE leurs propres gains/transactions.
- Les admins (service role) peuvent tout voir + importer les rapports.

À produire
1) SQL complet (Postgres) à exécuter dans Supabase SQL Editor :
   - Extensions utiles (uuid-ossp ou pgcrypto).
   - Types ENUM pour statuts.
   - Tables :
     a) public.sessions
     b) public.fan_transactions
     c) public.store_settlements (import rapports Apple/Google)
     d) public.celebrity_earnings (ledger net_final -> part célébrité)
     e) public.payout_batches
     f) public.payout_batch_items
   - Indexes clés (transaction_id/purchase_token, session_id, celebrity_id, period).
   - Triggers / functions :
     - fn_set_updated_at() pour updated_at.
     - trigger pour créer automatiquement une ligne fan_transactions à la création d’une session payante (optionnel).
     - function `public.compute_celebrity_earnings(period_start, period_end)` qui :
        * prend toutes les fan_transactions status='settled'
        * calcule `celebrity_share_cents` = round(net_final_cents * celebrity_revshare_bps/10000)
        * insère dans celebrity_earnings (idempotent : unique constraint sur fan_transaction_id)
   - Colonnes exactes demandées (voir ci-dessous).

2) RLS (Row Level Security) :
   - sessions :
     * fan (auth.uid() = fan_id) peut SELECT ses sessions
     * celebrity (auth.uid() = celebrity_id) peut SELECT ses sessions
   - fan_transactions :
     * fan peut SELECT ses transactions
     * celebrity peut SELECT les transactions liées à ses sessions
   - celebrity_earnings :
     * celebrity peut SELECT ses earnings
   - payout tables :
     * celebrity peut SELECT ses payout_batches + items
   - Seul service_role peut INSERT/UPDATE pour imports et statuts (via Edge Functions).

3) Modèle de statuts (important) :
   fan_transactions.status enum :
   - 'created' (ligne créée localement, pas encore validée)
   - 'store_confirmed' (achat validé via store API / webhook)
   - 'refunded' (refund/chargeback confirmé)
   - 'settled' (net_final importé depuis rapport financier)
   - 'included_in_payout' (affectée à un batch)
   - 'paid_out' (payée à la célébrité)
   store_settlements.status enum :
   - 'imported' (rapport importé)
   - 'applied' (net_final appliqué aux transactions)
   payout_batches.status enum :
   - 'created'
   - 'sent'
   - 'paid'
   - 'failed'

4) Colonnes minimales attendues :
- sessions:
  id uuid pk
  fan_id uuid (auth.users)
  celebrity_id uuid (auth.users)
  duration_minutes int
  scheduled_at timestamptz
  started_at timestamptz
  ended_at timestamptz
  status text/enum (draft/scheduled/completed/canceled)
  created_at, updated_at

- fan_transactions:
  id uuid pk
  session_id uuid fk sessions(id) unique (1 achat par session)
  fan_id uuid
  celebrity_id uuid
  platform text check in ('apple','google')
  product_id text
  gross_amount_cents int not null
  currency text not null default 'EUR'
  store_transaction_id text (Apple) nullable
  purchase_token text (Google) nullable
  store_order_id text (Google) nullable
  status enum
  store_confirmed_at timestamptz
  refunded_at timestamptz
  net_final_cents int nullable  -- rempli après import rapports
  net_currency text nullable
  settled_at timestamptz nullable
  celebrity_revshare_bps int not null default 7000  -- ex 70%
  created_at, updated_at
  unique constraints:
   - unique(store_transaction_id) where platform='apple'
   - unique(purchase_token) where platform='google'

- store_settlements:
  id uuid pk
  platform text
  period_start date
  period_end date
  report_source text (apple/google)
  file_name text
  imported_at timestamptz
  status enum
  raw_rows jsonb (optionnel)

- celebrity_earnings:
  id uuid pk
  celebrity_id uuid
  fan_transaction_id uuid unique
  period_start date
  period_end date
  net_final_cents int
  celebrity_share_cents int
  currency text
  created_at

- payout_batches:
  id uuid pk
  celebrity_id uuid
  period_start date
  period_end date
  total_share_cents int
  currency text
  status enum
  created_at, paid_at
  external_payout_ref text nullable

- payout_batch_items:
  id uuid pk
  payout_batch_id uuid fk payout_batches(id)
  fan_transaction_id uuid fk fan_transactions(id)
  celebrity_share_cents int
  created_at
  unique(payout_batch_id, fan_transaction_id)
